Assignment 03 Description:

  • Make the graphics interface platform-independent, only using Graphics.cpp to run on both OpenGL and D3D. This means all related graphics classes and methods used must have platform-independent interfaces.

  • Update the Mesh and Effect classes so that users can set up initialization data independently of the platform, rather than hardcoding it. Additionally, only retain the necessary data to minimize the memory of these classes as much as possible.

  • Using the updated interfaces, render two meshes, each bind a different effect.

How to make Graphics.cpp platform-independent

Adding the cView Class

I have created a new cView class to manage rendering views, including the depth-stencil view and render target view on the D3D platform. On the OpenGL platform, it is not necessarily required to record views separately, so currently on OpenGL, only a Clear () method for the default view needs to be implemented.

Figure 2. Class cView design

By implementing a platform-independent interface through the cView class, the corresponding differences between the Graphics.cpp files for the two platforms have been eliminated.

Adding the Present Interface to the sContext Structure

Due to the differences in how OpenGL and Direct3D submit rendering results, need to add a Present () method to the structure that manages the context. This method will provide an interface for submitting rendering results.

cEffect class design

A single effect takes up 8 bytes in OpenGL platform, takes up 48 bytes in D3D.

Figure 6. Class cEffect

For the D3D platform, during Bind(), the cShader needs to execute the direct3dImmediateContext->VSSetShader() function and the direct3dImmediateContext->PSSetShader() function.

While binding, only the m_shaderObject from cShader is needed. I attempted to retain only the Shader Object as a member variable of cEffect, as this would occupy less memory than storing the entire cShader. However, if I release the cShader object, the Shader Object loaded by that object would also become invalid. Therefore, I think currently have to keep the entire cShader object as a member variable since we need their information during Bind ().

For the OpenGL platform, once a program is created using a shader, the programId alone is sufficient to obtain the created shader information. It can be bound directly using programId, without needing to retain the loaded cShader data. As a result, the cEffect object’s memory usage is much smaller on the OpenGL platform.

CRenderState is needed for both platforms during Bind () to set up the rendering state.

cMesh class design

A single mesh takes up 16 bytes in OpenGL platform, takes up 32 bytes in D3D.

Figure 8. Class cMesh

During initialization, the vertex and index data will be uploaded to the GPU buffer, and the space will be allocated for the buffer. On the D3D platform, two ID3D11Buffer pointers are needed to record the addresses of the required buffers on the GPU, which are used to fetch the buffer data during Draw.

Similarly, on the OpenGL platform, three GLuint variables are used to access the VAO (Vertex Array Object), VBO (Vertex Buffer Object), and IBO (Index Buffer Object) created on the GPU.

When executing Draw on both platforms, the number of indices needs to be specified. However, on D3D, the starting index within the index buffer for drawing also needs to be specified. To avoid unnecessary memory alignment waste, this is declared externally.
