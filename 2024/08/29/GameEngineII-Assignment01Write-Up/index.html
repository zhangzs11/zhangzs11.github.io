Assignment 01 Description:

  • Add the Graphics module to an engine that lacks the Graphics component, establish dependencies, and add the Graphics module as a dependency to other engine modules that require it, enabling the Example Game project to run.

  • Read and understand how the Example Game operates, create MyGame project, and make it run successfully by adding the necessary dependencies. MyGame includes BuildMyGameAssets, which is responsible for compiling shaders, and the MyGame game project that loads the compiled shaders.

  • Read and understand the game runtime framework, modify the window title and game icon, and add log messages for initialization and cleanup of the window.

  • Read and understand the shader compilation and invocation framework, modify the fragment shader (pixel shader), and understand the concept of game simulation time to change the color of the triangle from the default white to a color that changes over time.

  • Read and understand user input handling, add an input event so that when the user presses a key (I used the spacebar), it changes the simulation time rate, making the triangle color change faster.

  • Optimize the declaration of constant buffers; store repeated declarations in a single file that all shader files can reference.

Analysis of Graphics Project References

  • Application project need reference to the new Graphics project.

  • The code in the Graphics section uses the Graphics namespace but didn’t require adding a reference. This is because it only declares variables, enumerations, and functions within the Graphics namespace without calling them.

Evaluation of Engine Code Structure and Style

Regarding Engine Code Organization:

I think the engine is well-organized, with each module being relatively independent, avoiding unnecessary dependencies. For instance, the rendering-related modules are only called within the Application module.

Regarding Code Style:

The code style is also very clear and well-defined, including file naming, variable naming, parameter naming, and comments. This clarity minimizes any potential misunderstandings I might have while reading the code.

Confusion:

Initially, I was confused about the purpose of BuildMyGameAssets and why it was necessary to compile shader files separately. Later, I realized that the compiled shader files are stored in the temp folder, and MyGame loads the compiled shader files at runtime instead of compiling them on the fly. This approach reduces runtime overhead and is more suitable when shaders do not change. When modifying MyGame, only need to recompile the game project.

Expectations of the Class:

I hope to learn about game engine development, including the design of architecture, setting up rendering pipelines, and knowledge of game engine optimization. Additionally, I’m interested in understanding how a complete engine coordinates its various components to work together seamlessly.

Optional challenges

Reducing Code Duplication in Shader Constant Buffers

I created a CommonBuffers.shader file and placed it in Engine/Content/Shaders/Common to store shared constant buffers. However, to ensure all shaders can include this file, it should be in the Engine/Content directory.

All shader files include it using the directive: #include <Shaders/Common/CommonBuffers.shader>.

Implementing Key Controls for Slow Motion

By adding a keyboard event in UpdateBasedOnInput(), I check if the Space key is pressed. If it is, I reset the SimulationRate to a higher value to speed up the animation.

When the Space key is released, the else branch of the keyboard event is executed, setting the SimulationRate back to its default value, thus restoring the animation to its normal speed.
