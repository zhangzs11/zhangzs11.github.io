Game Overview

My final game is inspired by the classic “Red Light, Green Light” game. The gameplay centers around a monster and the player, creating a tense and engaging experience.

Objective:

The player’s goal is to reach the destination while avoiding being caught moving by the monster.

Gameplay Mechanics:

  • The monster alternates between facing away from the player and facing toward the player.
  • When the monster faces away, the player can press W to move forward.
  • When the monster faces the player, the player must remain completely still and not press any keys.
  • Moving while the monster is watching results in failure.

Victory Condition:

The player wins by reaching the destination without being caught.

Failure Condition:

The player loses if they move while the monster is facing them.
This project was particularly interesting to me because it required careful integration of mechanics, visuals, and audio to create an immersive experience. I chose this concept because it combines suspenseful gameplay with clear, easy-to-understand rules, making it both challenging and fun.

Experience Developing the Game

Integration with My Engine

The game fully utilizes the engine I developed during the semester. My engine handles rendering all game objects in the scene, including the player and the monster. The monster’s rotation is achieved using the rigidbody system, while the player’s movement and interactions rely on the engine’s input and update systems. Additionally, I implemented various graphical effects, such as a skybox, post-processing effects for victory and failure, lighting, and shadow rendering.

Previous Choices and Their Impact

Easier Development:

Implementing support for 32-bit indices in earlier assignments allowed me to load more complex models into the game without limitations, making model rendering more straightforward.

Future Improvements:

While the current graphics engine design proved efficient and scalable for this project, I see potential for further enhancements. For instance, I could add more advanced graphical techniques, such as cascaded shadow mapping, anti-aliasing, or support for deferred rendering pipelines.

Experience Working with a Classmate’s System

I integrated the Audio System into my game to enhance the atmosphere. It provided background music and specific audio cues for victory and failure states. The integration was straightforward, as the interface was intuitive and easy to use. It relies on older MCI interfaces, which lack support for modern audio formats. It doesn’t support positional audio effects, which could be useful for scenarios like 3D FPS games where sound localization is critical.

What I Learned During the Semester

This semester’s assignments were incredibly valuable for understanding game engine design and development. I gained insights into:

Efficient Resource Management:

Learning how to optimize memory usage, such as choosing between 16-bit and 32-bit indices for meshes based on model complexity and minimizing unnecessary data transfers during runtime.

Data Pipelines:

Understanding the process of converting modeling data (e.g., from Maya) into binary formats optimized for runtime performance.

Graphics Programming:

Implementing essential rendering techniques like framebuffer management, lighting, and shadow effects.
These assignments provided practical experience in tackling real-world game engine development challenges, deepening my understanding of both the technical and architectural aspects of game design.

Personal Takeaways

This class has been a transformative learning experience for me. I’ve learned how to design a graphics module that integrates seamlessly into an engine and optimizes system resources for both GPU and CPU workloads. I also gained a better understanding of the role of data pipelines in game development.

One key realization was the importance of balancing simplicity and scalability in system design. For example, I initially implemented a single View class to handle all types of views in Direct3D (DSV, RTV, SRV). While this simplified the interface, it made the code harder to maintain and extend. I later refactored the design, creating separate classes for each view type. Although this introduced some repetition, it greatly improved clarity, maintainability, and extensibility.

Thoughts on Engineering, Architecture, and Design

I prefer designing systems “as you go,” focusing on the immediate problem and iterating to improve. While upfront design is helpful, it’s often impossible to anticipate all potential issues without practical implementation. Starting with a simpler solution allows for faster iteration and more effective problem-solving.

Characteristics of Good Software Architecture:

A good software architecture should:

  • Be intuitive for users to work with, offering well-designed interfaces with necessary functionality.
  • Be flexible and maintainable for developers, making it easier to extend and adapt the system over time.

Characteristics of Bad Software Architecture:

Conversely, a poor design might:

  • Overcomplicated simple tasks, confusing users or developers.
  • Be rigid, making future extensions or modifications difficult.

This course has helped me the balance between simplicity and scalability, teaching me to prioritize clear, maintainable designs while keeping an eye on future needs.
